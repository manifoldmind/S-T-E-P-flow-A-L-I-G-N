# S-T-E-P-flow-A-L-I-G-N

> *Black Box Drift Methodology*  
> *“No magic, just forming fairytale. No spells, just steps.”*

**Это методология осознанной разработки.**  
Она превращает хаос задачи в понятную последовательность: сначала выравниваем понимание, потом делаем маленькими шагами.  

Репозиторий создан для тех, кто:
- не хочет слепо копировать код из нейросетей;
- хочет использовать ИИ как учебник и наставника, а не «чёрный ящик»;
- ищет простой, но дисциплинированный способ разбирать любые задачи (от багов до новых фич).

---

## 🧭 Философия

Мы намеренно **замедляемся**.  
Прежде чем что‑то писать, мы отвечаем на вопросы: *Зачем? Какие ограничения? Какой подход? Что именно будем делать?*

Каждая задача проходит через **две фазы** — **A.L.I.G.N.** (анализ, обсуждение, подтверждение) и **S.T.E.P.** (исполнение маленькими шагами).  
Это не жёсткий шаблон, а каркас, внутри которого вы остаётесь главным архитектором решения.

---

## ⚙️ Как это работает

### Фаза 1: A.L.I.G.N. — выравнивание

Перед тем как писать код, мы **останавливаемся и фиксируем понимание**:

| Буква | Расшифровка                | Что делаем |
|-------|----------------------------|------------|
| **A** | **A**nalyze the task       | Называем основную **цель**, определяем её ценность в проекте |
| **L** | **L**ist constraints       | Формулируем условия, требования, ограничения |
| **I** | **I**dentify approach      | Обсуждаем инструменты, библиотеки, логику |
| **G** | **G**enerate breakdown     | Разбираем решение по шагам, объясняем зачем каждая строка/блок |
| **N** | **N**egotiate              | Сверяемся, получаем подтверждение — только после этого переходим к делу |

### Фаза 2: S.T.E.P. — исполнение

Когда понимание выровнено, идём по шагам, но не теряем осознанности:

| Буква | Расшифровка                | Что делаем |
|-------|----------------------------|------------|
| **S** | **S**tart with the smallest | Начинаем с самой первой осмысленной функции/блока — без неё ничего не заработает |
| **T** | **T**est minimally          | По возможности создаём минимальный тестовый пример / проверку |
| **E** | **E**xecute                 | Реализуем этот шаг, фиксируем результат |
| **P** | **P**roceed                 | Переходим к следующей задаче (или возвращаемся в A.L.I.G.N.) |

Эти фазы циклически повторяются для каждой логической единицы работы.

## 🚀 Как использовать

>[!tip] В идеале попробовать предварительно расписать свою версию :smiley_cat: (старание порождает _творчество_!)
1. **Прочитайте детальный алгоритм** → ссылка на `methodology/STEP-by-step.md` (или ваш файл).  
   Там расписано, как применять акронимы на реальных задачах.


3. **Возьмите шаблон** из папки `templates` (если она есть) или просто откройте новый markdown-файл и пройдите A.L.I.G.N. для своей задачи.

4. **Работайте по циклу**:  
   - A.L.I.G.N. → фиксируете в файле  
   - S.T.E.P. → реализуете один шаг  
   - Возвращаетесь к A.L.I.G.N. для следующего шага

5. **Если используете ИИ**, привлекайте его только на этапе **G** (Generate breakdown) или **T** (Test minimally) — как ускоритель, а не генератор готового ответа.

## 🤖 Работа с ИИ в рамках методологии

ИИ становится полезным именно тогда, когда вы уже **сформулировали задачу** в формате A.L.I.G.N.  
Примеры промтов:

- **На этапе G (Generate breakdown):**  
  *«У меня есть задача: [A]. Ограничения: [L]. Планируемый подход: [I]. Разложи решение на логические шаги с пояснением каждого.»*

- **На этапе T (Test minimally):**  
  *«Для шага [описание] напиши минимальный тестовый пример, который проверит корректность.»*

Так вы остаётесь в роли архитектора, а ИИ — в роли опытного помощника, который ускоряет, но не заменяет мышление.
